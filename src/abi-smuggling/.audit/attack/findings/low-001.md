# [LOW] Unprotected Initialization — Front-Running setPermissions

## Summary
`setPermissions()` has no access control beyond a one-time `initialized` flag. Any address can call it if they are the first caller, enabling a front-running attack during deployment that could hijack the vault's entire authorization policy or permanently brick it.

## Severity
- **Impact:** High — Could set malicious permissions granting attacker sweepFunds access, or brick the contract with empty permissions
- **Likelihood:** Low — Requires front-running the deployer's initialization transaction; in the test setup, deployer calls it atomically
- **Difficulty:** Low — Standard mempool front-running
- **Overall:** Low (high impact mitigated by low likelihood in practice)

## Prerequisites
- Vault must be freshly deployed and not yet initialized
- Attacker must observe the deployment transaction and front-run the `setPermissions` call
- Does NOT apply if deployment and initialization are atomic (e.g., via a factory contract)

## Affected Code
- **File:** `AuthorizedExecutor.sol`
- **Lines:** L25-L38
- **Function:** `setPermissions()`

```solidity
function setPermissions(bytes32[] memory ids) external {
    if (initialized) {
        revert AlreadyInitialized();
    }
    // ... sets permissions ...
    initialized = true;
}
```

## Root Cause
No `onlyOwner`, `msg.sender` check, or deployer address validation. The function relies purely on execution order (first caller wins).

## Attack Scenario
1. Deployer deploys `SelfAuthorizedVault` in transaction A
2. Deployer sends `setPermissions([...])` in transaction B
3. Attacker sees transaction B in the mempool
4. Attacker front-runs with `setPermissions([attackerSweepPermission])` granting themselves `sweepFunds` access
5. Deployer's transaction B reverts with `AlreadyInitialized`
6. Attacker calls `execute()` to sweep all funds

Alternative: Attacker calls `setPermissions([])` with empty array, permanently bricking the vault (TAG-006).

## Impact
- Attacker gains full control of vault authorization, or
- Vault is permanently bricked with no recovery path

## Recommendation
Add an access control modifier or pass the deployer address in the constructor:

```solidity
address public immutable deployer;

constructor() {
    deployer = msg.sender;
}

function setPermissions(bytes32[] memory ids) external {
    if (msg.sender != deployer) revert NotDeployer();
    if (initialized) revert AlreadyInitialized();
    // ...
}
```

Or combine deployment and initialization into a single atomic operation.
